## 1.2 ECS 架构
使用实体-组件-系统（**Entity-Component-System，简称 ECS**）架构处理主要逻辑，便于后续扩展。

### 概括
ECS 将类抽象为模板，对象抽象为实体，并将数据与逻辑分别分离到组件与系统中。系统仅关心拥有指定组件的实体。

### 思想
ECS 的核心思想是组合优于续承，示例：
```
续承:
基类，包含必要逻辑（血量移动等） <- 可近战攻击敌人 <- 可同时远程攻击与近战攻击的敌人 <- Boss

续承灵活性较低，组合一些逻辑需要多次续承，深度较高逻辑复杂

组合:
Boss(实体) 
	-> 近战攻击组件(存储伤害值、攻击速度)
	-> 远程攻击组件(存储发射的子弹、攻击速度、攻击范围)
	-> 血量组件(存储最大血量、当前血量)
	-> 移动组件(存储移动速度等)
	-> 更多组件...

系统处理 Boss 实体
	-> 近战攻击系统(根据组件存储的属性造成伤害)
	-> 远程攻击系统(根据组件存储的属性发射子弹、索敌)
	-> 血量系统(判断当前血量移除实体、处理即将受到的伤害)
	-> 移动系统(根据移动速度移动)
	-> ...其他系统由于没有其关心的组件不会处理当前实体
	
逻辑与数据分离，深度低逻辑简单，可任意组合
```

### 流程
```
组件 -> 模板 -> 实体 -> 系统
```

## 1.3 实体类型
根据实体的作用实体分为：
1. 普通实体，无特殊作用（敌人，友军）
2. （**damage**）伤害实体，仅用于造成伤害
3. （**bullet**）子弹实体，仅被其他实体发射（常被远程攻击发射）
4. （**modifier**）状态效果实体，依附于其他实体上持续影响实体（中毒、回血、燃烧）
5. （**aura**）光环实体，持续影响范围内其他实体（持续伤害范围内实体，持续为范围内实体回血）

## 1.4 需实现的功能
### 1.4.1 ECS 底层
#### 1. 通过组件创建模板
使用 JSON 存储组件与模板数据，后续写入用户目录，便于自定义

读取 JSON 并解析，允许通过格式化字符串输入常量与类型，允许模板之间的续承，并将模板存储在实体数据库（**entity_db**）

#### 2. 管理实体的创建与移除
将创建的实体统一存储到实体数据库 entity_db，用于后续系统遍历与索敌

实体不能在一帧的中间直接插入数据库，可能会导致时序问题，应使用队列在当前帧末尾统一插入实体。移除同理

思路：
1. 创建实体时将要创建的实体加入队列，并调用所有系统中的插入回调函数（钩子，实体插入时调用）
2. 当前帧末尾取出队列中的实体
3. 将实体插入实体数据库

#### 3. 系统主循环
遍历实体数据库中所有实体，调用所有系统的更新回调函数，并传入实体

系统更新回调函数需筛选拥有指定组件的实体

### 1.4.2 主要系统
#### 1. 实体系统
处理实体的脚本中的各种回调函数（更新、插入、移除时调用等）

#### 2. 血量系统
处理实体血量与将要受到的伤害，实体血量为 0 时移除实体

#### 3. 状态效果系统
处理状态效果的叠加、互斥等

#### 4. 光环系统
处理光环的叠加、互斥，更新回调等

#### 5. 导航路径系统
处理实体沿着路径移动

#### 6. 集结点系统
处理实体集结

#### 7. 子弹系统
处理子弹的飞行轨迹更新等，允许自定义飞行轨迹

#### 8. 兵营系统
处理兵营实体的生成实体

#### 9. 远程攻击系统
处理远程攻击的索敌、发射子弹等

#### 10. 近战攻击系统
处理近战攻击的拦截、攻击等

#### 11. 生成器系统
处理生成器实体的召唤实体

#### 12. 时间系统
统一时间管理，更新时间，将所有时间信息存入时间数据库（**time_db**）

#### 13. 精灵系统
处理精灵

#### 14. 关卡系统
初始化与更新关卡
